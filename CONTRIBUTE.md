# Руководство по внесению вклада

Данный репозиторий разработан преимущественно для сообщества [Remnawave](https://github.com/remnawave/panel).
Пожалуйста, учитывайте, что все правила, используемые или добавляемые в этот репозиторий,
будут использоваться всем сообществом [Remnawave](https://github.com/remnawave/panel), а не отдельными лицами.

> [!NOTE]
> Разработчики проекта «Kyreth | Proxy Cluster» уважают авторские права и считают своим
> обязательством указывать авторов каждого пользовательского набора правил.

## Пошаговая инструкция

Если вы используете клиент на основе ядра, компиляцию правил для которого поддерживает данный репозиторий,
и хотите поделиться своими наработками или создать новое публичное правило, выполните следующие действия:

1. Создайте форк (Fork) данного репозитория.
2. Клонируйте форкнутый репозиторий на локальную машину.
3. [Создайте новое правило](#создание-нового-правила), соблюдая [требуемый формат](#форматы-правил).
4. Зафиксируйте изменения и отправьте коммит (commit) в ваш форк.
5. Создайте Pull Request с вашими изменениями в основной репозиторий (Origin).

> [!IMPORTANT]
> Данная инструкция применяется исключительно к правилам, добавляемым вручную участниками сообщества.
> Правила, агрегируемые из внешних источников посредством автоматизированных регламентных заданий,
> могут иметь отличия от описанных ниже стандартов. Авторство таких правил указывается в [README.md](./README.md)
> документации репозитория

## Создание нового правила

Каждое новое правило должно соответствовать [определенному формату](#форматы-правил).
Хотя каждое ядро имеет свои специфические требования к формату файлов, их семантическая структура остается единой.

Правило для любого поддерживаемого ядра должно быть представлено в виде отдельной директории, содержащей ровно два файла:

1. `meta.yaml` — файл с мета-информацией о правиле.
2. `Исходный файл` — файл, из которого будет скомпилировано правило для ядра.

### Структура файла `meta.yaml`

Файл `meta.yaml` должен соответствовать следующей структуре:

```yaml
description: >
  Здесь должно содержаться исчерпывающее описание правила: его назначение, 
  особенности и целевая аудитория. Информация должна позволять участникам 
  сообщества однозначно понять суть и полезность данного правила.
author:
    username: Oidaho
    github: https://github.com/Oidaho
```

### Категории правил

Правила должны классифицироваться и размещаться в репозитории в соответствии с установленной семантикой.
В данном репозитории выделяются четыре основные категории:

- `geosite` — наборы доменных имен.
- `geoip` — наборы IP-адресов и подсетей.
- `packages` — наборы идентификаторов Android-пакетов.
- `processes` — наборы имен процессов Windows/Linux.

Например, если вы создаете правило для ядра Mihomo, содержащее набор доменных имен,
корректный путь размещения директории с правилом будет следующим:

```text
mihomo/geosite/...
```

### Именование правил

Имя правила должно четко и однозначно отражать сервис или категорию ресурсов, к которым оно относится.
Если однозначная привязка к конкретному сервису невозможна, следует использовать обобщающую категорию.

Примеры корректного именования:

- `geosite/telegram.list` — правило для доменных имен, используемых Telegram.
- `geoip/telegram.list` — правило для IP-адресов, используемых Telegram.
- `geosite/category-messengers.list` — правило для доменных имен различных мессенджеров.
- `geoip/category-messengers.list` — правило для IP-адресов различных мессенджеров.

### Форматы правил

#### Mihomo

Структура директории с правилом для Mihomo:

```text
my-new-rule/
├── meta.yaml
└── rule.list
```

Содержимое файла `rule.list` должно соответствовать спецификации ядра Mihomo.
Примеры можно найти в существующих правилах репозитория или в официальной [документации Mihomo](https://wiki.metacubex.one/en/).

#### Sing-Box

Структура директории с правилом для Sing-Box:

```text
my-new-rule/
├── meta.yaml
└── rule.json
```

Содержимое файла rule.json должно соответствовать спецификации ядра Sing-Box.
Примеры можно найти в существующих правилах репозитория или в официальной [документации Sing-Box](https://sing-box.sagernet.org/configuration/).

## Публикация правил

Процесс публикации пользовательских правил в репозиторий полностью автоматизирован и не требует ручного
вмешательства со стороны мейнтейнеров после принятия Pull Request.

Этапы публикации:

1. *Принятие изменений*: После успешного ревью и слияния (merge) вашего Pull Request в основную ветку репозитория, новые исходные файлы правил становятся частью кодовой базы.
2. *Автоматическая компиляция*: Специализированное регламентное задание (GitHub Actions), запускаемое **ежедневно в 00:00 UTC**, выполняет следующие действия:
    - Сканирует репозиторий на наличие новых или обновленных исходных файлов правил.
    - Компилирует их в бинарные форматы, необходимые для соответствующих ядер (Mihomo `.mrs`, Sing-Box `.srs` и др.).
    - Формирует итоговую структуру файлов в служебной ветке `release`.
3. *Создание релиза*: После успешной компиляции и обновления ветки `release` автоматически инициируется процесс создания нового GitHub Release с тегом `latest`. В этот релиз включаются все актуальные архивы с правилами.

> [!NOTE]
> Таким образом, ваше правило станет доступно для конечных пользователей в виде
> готового к использованию файла в ближайшем ночном обновлении репозитория.
